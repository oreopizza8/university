# 핸드오프 — M2 보안·데이터·통계 마일스톤 (2026-05-19 17:30)

## 이번 마일스톤이 해결한 것

PM 지시 3대 보안·데이터 과제 + 차기 마일스톤(크롤러·익명 통계·분포 시각화)을 한 번에 적재.

### A. 시드 데이터 더미 제거 + 외부 적재 경로 (SOP 3.3 Fallback 원칙 준수)

| 계층 | 위치 | 역할 |
|---|---|---|
| Fallback 시드 | `backend/src/main/resources/data/*.csv` | 시스템이 빈 DB로 절대 부팅되지 않게 하는 안전망 |
| 사용자 배치 | `backend/data-staging/*.csv` | 부팅 직후 비동기 UPSERT (대교협 70%컷 CSV, US CDS CSV) |
| 라이브 API | `KcueSyncService` | KCUE OpenAPI 키 있으면 부팅+일 단위 동기화, 실패 시 캐시 유지 |

- **Why 3계층:** 어느 한 채널이 끊겨도 진단 결과가 나가야 한다는 SOP Fallback 원칙. 더미 제거를 명령한 사용자 의도(=신뢰 가능한 결과)는 "시드 자체를 지우는 것"이 아니라 "운영 시 시드만으로 답변하지 않게 만드는 것" — 그래서 상위 계층이 항상 시드를 덮어쓰도록 우선순위를 줬다.
- **출처 추적 의무:** `data-staging/SOURCES.md`에 배치 시 출처·일자·라이선스 기록 의무. 미기재 데이터는 운영 배포 금지.

### B. Rate Limit (`RateLimitFilter`)
- Bucket4j 8.10.1 인메모리. IP 단위 분당 10 / 시간당 50 이중 버킷.
- `POST /api/diagnostic` 만 대상. GET 자동완성·통계는 부담 작아 제외.
- X-Forwarded-For → X-Real-IP → RemoteAddr 순으로 클라이언트 IP 추적 (리버스 프록시 호환).
- **Why 이중 버킷:** 분당만 두면 슬로우 폴링(50초마다 1회)으로 시간당 72회까지 가능 → 시간당 상한이 분당 상한을 받침. PM 지시 정확히 반영.
- **응답 형식:** 429 + `Retry-After: 60` + JSON 에러. 프론트는 그대로 표시.

### C. 검증 보강 (`DiagnosticRequest`)
- 모든 등급 필드: `@Min(1) @Max(9)`.
- 백분위 필드 (장차 점수 입력 모드용 슬롯): `@Min(0) @Max(100)`.
- 학년: `@Min(1) @Max(3)`.
- 문자열: 대학명·학과명·시험종류 모두 `@Size(max=...)`.
- 한글 에러 메시지 → `GlobalExceptionHandler`가 `fields` 맵으로 직렬화.

### D. 크롤러 (`KcueSyncService`)
- WebFlux WebClient + 30초 타임아웃. 실패 시 `onErrorResume` → Mono 빈 리스트 → 캐시 유지.
- 트리거: `ApplicationReadyEvent` (부팅 직후) + `@Scheduled(cron=...)` (기본 매일 04:30).
- 상태 노출: `GET /api/ingest/status` (lastStatus, lastSuccessAt), 수동 트리거 `POST /api/ingest/kcue/run`.
- **Why 부팅+스케줄 이중 발화:** 시즌 시작 직후 컨테이너 재시작이 잦은데, cron만 의존하면 컨테이너 수명보다 다음 트리거가 멀어 진단 결과가 시즌 앞 데이터로 채워질 위험.
- **API 키 미설정 시:** `SKIPPED_NO_KEY` 로그 1줄만 남기고 깔끔하게 패스 (개발 환경에선 키 없이도 빌드/실행 보장).

### E. 익명 통계 (`DiagnosticEvent`, `StatisticsService`)
- 진단 응답 후 `@Async("statsExecutor")`로 백그라운드 적재 → 사용자 응답 지연 0.
- 저장 컬럼: `mode / targetUniversity / targetDepartment / percentileBucket(5단위) / signal / createdAt`. **PII 없음.**
- 적재 실패 시 warn 1줄만 남기고 무시 — 통계가 본 응답을 절대 막지 않는다.
- **Why 5단위 버킷:** 진단 응답이 백분위 99.1, 99.3 식으로 잘게 쪼개지면 히스토그램이 가독성 0. 5단위로 미리 양자화해 저장 시점부터 집계 친화적.
- **Why 5단위 버킷 (저장 시 양자화):** 응답 시점에 미리 buckets 화하면 집계 쿼리가 `GROUP BY bucket` 한 줄로 끝남 → 분포 API가 O(n)에서 O(1 group) 수준으로 가벼워진다.

### F. 분포 시각화 (`DistributionChart`)
- Recharts BarChart. 같은 토글 모드 + 같은 대학·학과 표본만 노출(서버에서 필터).
- **본인 위치는 두 가지로 강조:** 해당 버킷 Bar는 짙은 네이비, 그 위에 빨간 점선 ReferenceLine "나".
- 표본 수·필터 조건을 헤더에 노출. 표본이 적으면 "분포는 흔들린다" 디스클레이머 자동 노출.
- **토글 연동 흐름:** 토글 → 진단 응답 mode → 차트가 같은 mode로 분포 재조회. 별도 토글 핸들러 없이 응답 데이터로만 흐름 일관.

## 새 엔드포인트

| Method | Path | 설명 | Rate Limit |

## 운영자 체크리스트

1. **KCUE 키 발급 → `backend/.env`에 `KCUE_API_KEY=...`** 채우기.
2. **데이터 배치:** `backend/data-staging/`에 CSV 3종(템플릿은 `data-staging/README.md` 참조).
3. **`SOURCES.md` 갱신** — 출처·일자·라이선스. 미기재 데이터는 배포 금지.
4. **Rate Limit 튜닝:** 실측 트래픽 보고 `.env`의 `RATELIMIT_PER_MINUTE/HOUR` 조정.
5. **익명 통계가 늘면** Postgres 마이그레이션 검토(현 H2 인메모리는 시연용).
