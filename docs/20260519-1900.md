## 2. 누적 의사결정 — Why 중심

### 2-1. 데이터 파이프라인을 3계층으로 둔 이유
```
L0 Fallback 시드 (resources/data/*.csv, 최소 안전망)
   ↑ 덮어쓰기
L1 사용자 배치 CSV (data-staging/*.csv, 부팅 시 비동기 UPSERT)
   ↑ 덮어쓰기
L2 라이브 API (KcueSyncService, 부팅 + 일 cron)
```
- **단일 출처(API만)는 위험:** KCUE 다운/스키마 변경 시 전체 마비.
- **단일 출처(CSV만)는 stale:** 시즌 변화 추적 불가.
- **단일 출처(시드만)는 더미:** 실데이터 대체 안 됨.
- 3계층을 두면 어느 채널 1개 끊겨도 진단 응답이 비지 않는다. SOP 3.3 Fallback 원칙의 충실한 구현.

### 2-2. SAT 환산식을 `400 + (합/200)×1200` 으로 고정한 이유
- 백분위 합 0 → SAT 하한 400, 합 200 → SAT 상한 1600. **CollegeBoard 공식 스케일과 정확히 일치.**
- 수식 한 줄로 끝나 디버깅·역추적·사용자 신뢰도 검증이 모두 가능.
- 정확도보다 **재현성·설명가능성**을 우선 — 본 도구는 예측기가 아니라 위치 확인기이기 때문.

### 2-3. 신호등을 "범위(40~60%)" + 격차 기반으로 한 이유
- 단일 숫자 표기("50%")는 만능 예측기 오인을 부추긴다 → 도구 목적 자체와 충돌.
- 격차 -5pp 이내 안정 / -15pp까지 소신 / 그 이하 상향. 정시 컷이 해마다 ±5pp 출렁이는 자연 노이즈를 경계로 사용.
- 사용자가 점수 1점 차로 신호가 바뀌는 짜증을 줄이고, 도구의 보수성을 노출.

### 2-4. Rate Limit 을 분/시 이중 버킷으로 한 이유
- 분당 상한만 있으면 슬로우 폴링(50초마다 1회) 시 시간당 72회 우회 가능.
- 시간당 상한을 추가해 봉인. Bucket4j 가 `addLimit` 누적 호환이라 1줄 추가로 끝.
- 인메모리 선택 이유: 멀티 인스턴스 운영 전까지 Redis 부담을 만들지 않는다.

### 2-5. 익명 통계를 5단위 백분위 버킷으로 양자화한 이유
- 진단 응답을 그대로 저장하면 백분위 99.1, 99.3, 99.5 식 fingerprint 가 만들어져 **단일 사용자 역추적 가능성** 발생.
- 5단위 양자화 시 동일 학과·동일 버킷에 다수가 모이므로 익명성 강화 + 집계 쿼리도 `GROUP BY bucket` 단일 패스로 경량.
- 회수 정책은 90일 cleanup 스케줄러로 보강 예정(현재 TODO).

### 2-6. 부팅+cron 이중 발화 (KcueSyncService)
- 컨테이너 수명이 cron 주기보다 짧을 때 cron만 의존하면 시즌이 시작돼도 한참 동안 시즌 앞 데이터가 노출됨.
- 부팅 직후 1회 + 일 단위 cron 둘 다 실행해 stale 윈도우 최소화.

### 2-7. xlsx 처리에 Apache POI 대신 Python 1회 변환기를 선택한 이유
- POI 영구 의존 시 JAR 5~10MB 증가 + 빌드 시간 + 런타임 메모리.
- 학과 마스터 갱신 주기는 분기/연 단위 → 영구 의존 가치 < 변환 1회 비용.
- 사용자 환경에 Python 3.14 + openpyxl 가용 → 변환기는 staging 폴더에 1회용으로 남기고 Java는 깨끗한 CSV만 처리.

### 2-8. 폐지·모집중단 학과 사전 필터링 이유 (55k → 28k)
- 자동완성에 폐지 학과가 떠오르면 사용자 혼란.
- 인메모리 H2 부팅 시간·메모리 절약.
- 데이터 신뢰성 = "현재 모집 중인 학과만 본다"는 명확한 약속.

### 2-9. .env / .env.example 인시던트 대응
- 2026-05-19 인시던트: 사용자가 키를 `.env.example` 에 직접 입력 → 커밋되면 GitHub 노출.
- 즉시 `backend/.env` 로 이전(gitignored), 템플릿은 placeholder로 복원.
- `.gitignore` 에 `backend/.env`, `frontend/.env` 명시 추가.
